Agis en tant qu'ingénieur Full-Stack spécialisé en systèmes mécatroniques pour concevoir une interface de commande Web (IHM) hautes performances, pilotée par WebSerial, pour un traceur de plans cartésien.

Contexte Technique :

Le système est un traceur de plans à trois degrés de liberté ($X, Y, Z$) piloté par un microcontrôleur Arduino Mega 2560 via une liaison série UART (115200 bauds).

Le déplacement dans le plan horizontal repose sur une architecture CoreXY (ou H-Bot, plus précisément H8). 
- Actionneurs : 2 moteurs pas-à-pas Nema 17 (200 pas/tr).
- Transmission : Courroies crantées GT2 (poulies 20 dents, pas de 2 mm).
- Modèle de couplage : Les mouvements résultants (X, Y) sont une combinaison linéaire des rotations moteurs (M1, M2) :
  - \Delta x = \frac{R}{2}(\Delta\theta_1 + \Delta\theta_2)
  - \Delta y = \frac{R}{2}(\Delta\theta_1 - \Delta\theta_2)
- Résolution : 0,2 mm par impulsion moteur.

L'axe Z gère la levée du stylet pour alterner entre les phases de tracé et de déplacement rapide.
- Actionneur : 1 moteur Nema 17.
- Transmission : Système vis-écrou.
- Compliance : Montage hybride incluant une solution de compliance passive (bandes élastiques) pour absorber les chocs et compenser les irrégularités de planéité du support.

Le micrologiciel (firmware) intègre un pipeline de traitement déterministe :
- Interpolation : Algorithme DDA (Digital Differential Analyzer) exécuté dans le référentiel moteur pour garantir la rectitude des vecteurs et supprimer le jitter temporel.
- Dynamique : Profil de vitesse trapézoïdal avec une vitesse initiale plancher (V_min = 20 RPM) pour franchir les zones de résonance mécanique.
- Sécurité : Surveillance active via 3 capteurs de fin de course (limit switches) et bornage logiciel (soft limits) via la fonction `constrain`.
- Communication : Protocole de commande textuel structuré (ex: `x100y50v60`) géré de manière asynchrone.
- Drivers utilisés : TB6560.
- Zone de traçage : Elle mesure 400x400mm. En revanche, quelques modifications post-fabrication peuvent être réalisées. On doit être capable de changer ce paramètre facilement depuis le backend (réfère toi aux constantes/paramètres contenues dans le code Arduino en pièce jointe).
- Déplacement maximal selon Z : 100mm. Encore une fois, c'est sujet à modifications. Réfère-toi à la constante présente dans le code Arduino.

L'objectif est de minimiser drastiquement la distance parcourue avec le stylo levé (Pen Up) et de réduire le temps total d'usinage.

PARTIE SÉCURITÉ ET FIABILITÉ SYSTÈME :
- Protocole Handshake (Critique) : Pour éviter la saturation du buffer de réception de l'Arduino, l'envoi de données doit être strictement conditionné. Après l'envoi d'une ligne d'instruction (ex: `x100y50v60\n`), le JS doit obligatoirement attendre la réception de la chaîne exacte `OK` depuis l'Arduino avant de libérer la ligne suivante.
- Interruption Matérielle : Si la console reçoit un message commençant par `LIMIT` (déclenchement physique d'un capteur), l'interface doit instantanément basculer en état d'erreur (ERROR), purger la file d'attente et émettre une alerte sonore.
- Machine à États : Implémenter un gestionnaire d'états strict (`IDLE`, `HOMING`, `RUNNING`, `ERROR`). Tant que le Homing initial n'a pas été exécuté et validé par l'Arduino, toutes les commandes de déplacement manuel et le lancement d'un tracé doivent être désactivés (visuellement grisés et non cliquables).
- Asynchronisme (Web Worker) : Le parsing et la discrétisation géométrique d'un fichier SVG lourd peuvent geler le navigateur. Ce traitement mathématique doit impérativement être déporté dans un Web Worker fonctionnant en arrière-plan.
- Calibrage Spatial :
  - Fournir une option (case à cocher) pour inverser l'axe Y lors de l'importation du SVG.
  - Intégrer un sélecteur d'origine physique (Top-Left, Bottom-Left, Center) pour s'adapter à la position de la feuille de papier sur le plateau.
- Procédure "Calibration Z" : Ajouter un mode spécifique permettant de faire descendre le stylo en mode pas-à-pas (Jog) jusqu'au contact avec le papier. Une fois la position validée par l'utilisateur, l'enregistrer dans le `localStorage` sous la clé `Z_PROBING_ZERO`.
- Résilience aux Déconnexions : Surveiller l'état du port série. En cas de fermeture inattendue (câble arraché), mettre virtuellement le tracé en pause, sauvegarder l'index du dernier vecteur envoyé avec succès, et afficher un bouton "Reprendre le tracé (Resume)" qui sera actif dès la reconnexion du port.
- Pre-Flight Checklist : Avant de déverrouiller le bouton "Start" pour un tracé, afficher une modale interactive avec des cases à cocher : "Moteurs sous tension ?", "Stylo monté ?", "Origine dégagée ?". Le bouton ne s'active qu'une fois la checklist complétée.
- Détection de connectivité : Implémenter un script de surveillance de l'état du réseau (via `navigator.onLine`). Si l'utilisateur lance l'interface sans connexion internet (usage local/hors-ligne) :

### Spécifications de l'API Série pour l'Interface Graphique (IHM)

**Configuration matérielle :**
*   **Baudrate :** 115200.
*   **Terminaison :** LF (`\n` ou `\r\n` toléré car la chaîne est nettoyée par `.trim()`).
*   **Protocole :** Handshake implicite strict. L'IHM ne doit jamais envoyer une commande de mouvement tant qu'elle n'a pas reçu le signal de fin de traitement (`OK` ou une erreur) de la commande précédente pour éviter la saturation du buffer RX.

**Commandes de contrôle (Système) :**
*   `s` : Requête de statut. Renvoie la position absolue actuelle (`>> X=...`) suivie de `OK`.
*   `i` : Séquence Homing XY. Séquence bloquante. Renvoie `OK` (succès) ou `LIMIT X`/`LIMIT Y` (échec palpage).
*   `k` : Séquence Homing Z. Séquence bloquante. Renvoie `OK` ou `LIMIT Z`.
*   `a` ou `A` (Arrêt d'urgence) : Seule commande scrutée de manière asynchrone (via `Serial.peek()`) au sein même des boucles d'exécution temporelles. Interrompt la trajectoire instantanément. Renvoie `!!! ARRET !!!` suivi du statut positionnel.
*   `p` : Éditeur de paramètres interactif. **Avertissement architectural :** Cette commande ouvre un sous-menu bloquant nécessitant plusieurs échanges séquentiels avec un timeout de 10s. Pour une IHM de production, il est fortement déconseillé de l'utiliser à moins d'implémenter une machine d'état spécifique côté client. Les paramètres devraient idéalement être envoyés en une seule trame formatée.

**Commandes cinématiques (Déplacements par Deltas) :**
*   `x[val]y[val]v[val]` : Déplacement XY. `val` est un flottant (ex: `x50y-20.5v80`).
*   `x[val]v[val]` ou `y[val]v[val]` : Déplacement d'un seul axe.
*   `z[val]v[val]` : Déplacement Z.

**Analyse des signaux de retour (Parsing) :**
*   `OK` : Exécution terminée, attente de la trame suivante.
*   `ERR` : Rejet syntaxique (ex: absence de coordonnée, ou `v` manquante) ou paramètre aberrant ($V \le 0$ ou $\Delta = 0$).
*   `LIMIT [AXE]` : Signal critique de sécurité. L'IHM doit le capter pour figer ses envois, alerter l'utilisateur et forcer une nouvelle procédure de prise d'origine.
*   `>> X=[val]mm Y=[val]mm Z=[val]mm  [0-400 / 0-100mm]` : Chaîne de télémétrie pure. Prévoir un parser Regex pour extraire dynamiquement les coordonnées spatiales exactes.
